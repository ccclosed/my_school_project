# Критичные баги найдены и исправлены ✅

## 🔴 КРИТИЧНЫЕ (могли привести к крашу)

### 1. ISR8 (Double Fault) - Переключение стека после push
**Проблема:** 
```asm
isr8:
    push 8
    push r15
    ...
    lea rsp, [df_stack_top]  // ❌ Переключаем стек ПОСЛЕ push!
```

**Последствия:** Затирание стека, потеря данных, невозможность восстановления.

**Исправление:**
```asm
isr8:
    lea rsp, [df_stack_top]  // ✅ Переключаем стек СРАЗУ
    push 8
    push r15
    ...
```

Double fault теперь никогда не возвращается (правильное поведение).

### 2. ISR13/ISR14 - Неправильная обработка error code
**Проблема:**
```asm
isr13:
    push 13              // ❌ Пушим номер исключения
    push r15             // CPU уже запушил error code!
    ...
    add rsp, 16          // ❌ Неправильный размер
```

**Последствия:** Стек рассинхронизирован, iretq вернется на неправильный адрес → triple fault.

**Исправление:**
```asm
isr13:
    push r15             // ✅ Не пушим номер
    ...
    mov rdi, [rsp + 15*8]  // ✅ Читаем error code из стека
    call gpf_handler
    ...
    add rsp, 8           // ✅ Попаем только error code
```

Обработчики теперь получают error code как параметр.

---

### 3. Scheduler - Неправильное выравнивание стека
**Проблема:**
```rust
sp.offset(-2).write(stack_top);  // ❌ RSP выровнен на 16
```

**Последствия:** При вызове функции из задачи стек будет выровнен неправильно → SSE инструкции крашнутся.

**Исправление:**
```rust
sp.offset(-2).write(stack_top - 8);  // ✅ RSP misaligned на 8
```

x86_64 ABI требует: перед `call` стек должен быть misaligned на 8 байт (потому что `call` пушит return address).

## ⚠️ ВАЖНЫЕ (могли привести к некорректной работе)

### 4. CPUID - Использование зарезервированного RBX
**Проблема:**
```rust
out("ebx") ebx,  // ❌ RBX зарезервирован LLVM
```

**Последствия:** Компилятор отказывается компилировать.

**Исправление:**
```rust
"mov {tmp}, rbx",
"cpuid",
"xchg {tmp}, rbx",
tmp = out(reg) ebx,
```

Используем временный регистр для обхода ограничения.

---

### 5. GDT - Неправильные лимиты для 64-bit сегментов
**Проблема:**
```rust
GdtEntry::new(0, 0, ACCESS_CODE, GRAN_4K_64),  // ⚠️ limit=0
```

**Последствия:** В 64-bit режиме base/limit игнорируются, но это плохая практика.

**Статус:** Оставлено как есть (работает, но можно улучшить).
